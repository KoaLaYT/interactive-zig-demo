# Interactive Zig Demo

A Zig rewrite of [Interactive Programming in C](http://nullprogram.com/blog/2014/12/23/), a demo of [interactive programming](http://en.wikipedia.org/wiki/Interactive_programming).

### How to run

`make run` to start the game, then edit `game.zig`, `make lib` to see the update.

Controls: `r` to randomize, `q` to quit.

### What This Project Actually Does

It's a terminal-based Conway's Game of Life that supports **hot-reloading**. The main executable loads game logic from a dynamic library (`libgame.dylib`) at runtime via `dlopen`. Every loop iteration, it checks if the library file's inode has changed -- if so, it swaps in the new version on the fly. You edit code, run `make lib`, and your changes appear instantly without restarting the game.

The architecture is clean: `main.zig` is the loader, `game_api.zig` defines the shared interface (`Api` and `State` as `extern struct`s with C calling convention), and `game.zig` contains the actual game logic and rendering via ncurses.

### What's Good

- **The hot-reload architecture is elegant.** Inode-based change detection, proper `extern struct` for ABI stability, function pointers through a single exported `api` symbol -- this is the right way to do it. The deliberate comment in `build.zig` ("DO NOT LINK HERE, we use dlopen") shows intentional design.
- **Double-buffering is correctly implemented.** Two cell grids with a `select` flag for flipping. The `iterate()` reads from one buffer and writes to the other. No off-by-one errors in the neighbor counting. Toroidal wrapping via `@mod(ix + dx + iw, iw)` is correct.
- **Clean separation of concerns.** The loader knows nothing about game logic. The library knows nothing about loading. The API contract sits in a shared file. Textbook plugin architecture.
- **Proper resource management.** Deferred `deinit` calls, proper allocator passing through `extern` boundaries via pointer casting with `@alignCast()`.

### What's Not So Good

- **macOS only.** The library path is hardcoded to `.dylib`. On Linux it should be `.so`. On Windows, this entire approach would need a rewrite. For a demo, this is forgivable. For anything beyond that, it's not.
- **`unreachable` instead of error handling.** `alloc.alloc(u8, ...) catch unreachable` in `game.zig` -- if allocation fails, the program panics with no useful message. In a demo, sure. In practice, handle the error or at least `@panic("out of memory")` with a message.
- **Fixed PRNG seed.** `std.Random.DefaultPrng.init(42)` means the "random" initialization is the same every run. Either use a time-based seed or make 42 a deliberate easter egg (document it).
- **No signal handling.** If you Ctrl+C, ncurses doesn't get its `endwin()` call. Your terminal will be in a broken state. A `SIGINT` handler that cleans up would be a nice touch.
- **No cross-platform build logic.** `build.zig` could use `@import("builtin").os.tag` to select the right library extension and linker flags. Zig makes this easy -- might as well use it.
